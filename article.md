# 目的

煮饭还是叫外送前，可以看一下什么生肉没看过，打个指令过几分钟就能在电视或 iPad 上配饭看。

所以很讲求一次性自动化的翻译品质。

# 缘由

年初 2025 M1 熟肉苦等不到，想说 AI 翻译现在也算在价格与能力的甜蜜点，就试著做了个自动化，感觉大概可以做到 70 ~ 80 分左右的品质。

使用到目前三个月，觉得差不多没什么痛点了，于是分享纪录一下。

# 流程

简单来说就四个步骤：**下载**、**语音辨识**、**翻译(预处理)**、**翻译(批次)**。

## 下载与资讯撷取

YT-DLP 配上科学上网

![](doc/article2.png)

另外，tver 跟 abema 可以从 API 拿到节目资讯以及出演人员，尤其出演人员很关键，会有名称、所在组合、节目中的角色等重要讯息。

## 语音辨识

全部用下来觉得最好的是 ElevenLabs 的 `Scribe v2`。

接下来是 豆包语音 的 `音视频字幕生成`，然后 阿里云 的 `Fun-ASR` (注意不是 `fun-asr-mtl`)。

其他 Google 的 `Chirp 3` 还是 OpenAI 的 `GPT-4o-transcribe `等都是不合格的。而自架方案也都试过了，就算是最新的 `Qwen3-ASR` 都算不上能用。

## 翻译

模型选择上，也基本都试过一轮，最后还是 `Google Gemini 3` 翻译最好，日翻中比英翻中难太多了，尤其很多氛围意境很难单纯从字幕上直接获取。

网路上大多数都在比 agent 调用、比写 code、比 tool calling，但日翻中很难从这些角度去评价出来。

### 翻译(预处理)

这个步骤，会将以下资料喂给 LLM：

1. 语音辨识出来的日文字幕
2. 节目的完整音档
3. 节目的多个截图
4. 节目资讯 (下载步骤获取的节目标题、介绍、出演人员资讯)

先产生出一个节目概要资讯当作绝对事实，之后分段批次翻译使用，以免出现资讯不同步的状况。

Gemini 3 在有原本音档跟图片辅助下，翻译的品质我觉得是跨过「你翻的这些我都看得懂，配上画面我却看不懂他们在聊什么」这种常见 AI 翻译困惑的关键。

### 翻译(批次)

接下来会采取分段同时翻译，通常是以 6000字 (约五分钟) 一段。

为什么会采取分段？主要有两个优点：

1. 速度，多段平行翻译，速度当然也比较快。
2. 品质，减少上下文的 Token，会大幅增加翻译品质，LLM 的思考 token 也能完全专注在这五分钟发生什么。格式或时间轴错误也会减少。

但也因此，需要先预处理的概要资讯去维系各段翻译的一致性。

最早有使用 `Gemini 3 Pro` 直接一次全部翻译，但结构错误机率高，大概到二十分钟后时间轴就有可能出现偏移。

这步骤喂给 LLM 的资料：

1. 该段的切割音档
2. 该段的多个图档 (每 30 秒一张)
3. 预处理的概要资讯
4. 该段的日文字幕

# 总结

目前流程处理一个小时的节目大概是 20 元台币，耗时 10 ~ 15 分钟左右。

不过其实瑕疵还是很多，譬如：

1. 特殊事件或专有名词，没办法另外在顶部加注释。
2. 大喜利的翻译品质。
3. 需要对电波的漫才翻译品质。
4. 有时候甚至会自作聪明做修正，像某段在讨论 M1 冠军，直接把 Takuro 改翻成 令和浪漫。
